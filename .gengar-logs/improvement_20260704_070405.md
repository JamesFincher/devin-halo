## Current Behavior

The verifier (Step 7 of halo-build.md) currently returns a text-based verdict: APPROVED, REJECTED, or NEEDS_REVISION. This is a vibe-based assessment — the verifier reads the code, forms an opinion, and writes a report. There is no machine-verifiable artifact:

1. **No structured evidence**: The verifier report is freeform markdown — no typed fields, no hashes, no exit codes.
2. **No certificate chain**: There's no proof that tests actually ran and passed. The verifier *says* "test suite: 12 passing" but nothing prevents a malfunctioning verifier from saying that when tests actually failed.
3. **Stop Hook is missing**: The workflow's Step 11 (commit and update) runs unconditionally after the verifier says APPROVED — there's no gate that checks "did the verifier actually produce verifiable proof?"
4. **No attestation**: No evidence that the git diff matches acceptance criteria, that no denylist paths were touched, that the test runner actually executed.

This is the exact same pattern our L20 (evidence.py) fixed in the Hermes harness. The Loop Engineering Run #77 proved: vibe-based stop gates produce false positives because there's no structural difference between "verifier passed" and "verifier says it passed."

## Desired Improvement

Replace vibe-based verifier output with **typed evidence certificates**. Each certificate is a machine-verifiable statement:

```
CERT(type=TEST_SUITE_PASS, runner=pytest, tests=12, passed=12, exit_code=0, hash=sha256:abc123)
CERT(type=DIFF_CLEAN, changed_files=["src/feature.ts"], denylist_touched=false, hash=sha256:def456)
CERT(type=ACCEPTANCE_MET, criterion="Email validation rejects invalid formats", 
     evidence="src/feature.test.ts:22-25 tests invalid email ", hash=sha256:ghi789)
CERT(type=BUILD_OK, command="npm run build", exit_code=0, hash=sha256:jkl012)
```

The Stop Hook in halo-build.md checks: ALL required certs present AND each cert's hash matches re-verification. If any cert is missing or its hash doesn't recompute, the loop halts.

This is the "No Certificate, No Execution" principle from the PCE architecture (arXiv:2605.24462) applied to the Devin Halo build loop.

### Change 1: Add Certificate Format to halo-verifier.md (line 78-103)

### Step 8: Produce Verification Report

VERIFICATION REPORT

VERDICT: APPROVED / REJECTED / NEEDS_REVISION
```

AFTER:
```
### Step 8: Produce Evidence Certificates

Generate typed evidence certificates. Each certificate is a single line with format:
CERT(type=<TYPE>, status=<PASS|FAIL>, details=<JSON-kv-pairs>, hash=<sha256>)

Required certificate types:
1. **TEST_SUITE_PASS** — full test suite ran and all passed
   CERT(type=TEST_SUITE_PASS, status=PASS, tests=12, passed=12, exit_code=0, hash=<sha256 dump>)
2. **DIFF_ANALYSIS** — changed files reviewed, no denylist, no unrelated changes
   CERT(type=DIFF_ANALYSIS, status=PASS, files_changed=3, denylist_touched=0, hash=<sha256 dump>)
3. **ACCEPTANCE_CRITERIA** — per-criterion PASS/FAIL
   CERT(type=ACCEPTANCE_CRITERIA, status=PASS, met=4, total=4, hash=<sha256 dump>)
   CERT(type=ACCEPTANCE_CRITERIA, status=FAIL, met=2, total=4, failed_criteria="Email validation", hash=...)
4. **CODE_QUALITY** — code quality assessment
   CERT(type=CODE_QUALITY, status=PASS, conventions=yes, security=ok, hash=<sha256 dump>)

The hash is sha256sum of the FULL certificate string (excluding hash field itself).
This makes each certificate tamper-evident: you can recompute the hash to verify.

After ALL certificates, output:
VERDICT: APPROVED  (only if ALL required certs are PASS)
VERDICT: REJECTED  (if any cert is FAIL)
VERDICT: NEEDS_REVISION  (if borderline but fixable)

STOP_GATE_TOKEN: cert_count=4, all_pass=true
```

### Change 2: Add Stop Hook to halo-build.md Step 7 (after line 107)

Add after "Returns: APPROVED, REJECTED, or NEEDS_REVISION":

```
After the verifier returns, the build loop MUST:
1. Read ALL certificate lines (CERT(...)) from verifier output
2. Verify each certificate's hash: recompute sha256 of the cert (without hash field), confirm match
3. Confirm STOP_GATE_TOKEN reads: cert_count matches actual cert count, all_pass=true
4. Only if all certs verify AND all pass → proceed to Step 9
5. If any cert fails hash verification → ABORT, escalate: "Evidence certificate tampered or corrupted"
6. If any cert status is FAIL → treat as REJECTED, proceed to Step 8's rejection path
```

This goes between the current "Returns: APPROVED, REJECTED, or NEEDS_REVISION" and "### Step 8: Handle Verification Result".

### Change 3: Accepting Edits in halo-build.md Step 11 (line 138-145)

Replace the unconditional STATE.md update with a cert-verified update:

BEFORE:

### Step 11: Commit and Update State
- `git add -A`
- Commit: `feat: SXXX — <story title>`

### Step 11: Commit and Update State (Gated by Evidence Certificates)

- Confirm STOP_GATE_TOKEN certs are still valid (hashes match)
- Generate a DEPLOYMENT_CERT: CERT(type=DEPLOYMENT, story=SXXX, status=<verdict>, commit=<sha>, hash=...)
- Attach DEPLOYMENT_CERT to the commit message as a git trailer:
  ```
  feat: SXXX — <story title>

  ```
- `git add -A`
- Commit: `feat: SXXX — <story title>`
- Update `STATE.md`:
```
